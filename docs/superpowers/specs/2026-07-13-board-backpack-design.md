# 《年糕食肆》棋盘背包设计

## 目标

将背包从棋盘外的底部按钮改为棋盘内的特殊功能物件。背包占用一个普通棋盘格，玩家可以根据自己的习惯拖动它，同时保留背包作为功能入口的识别度和操作安全性。

## MVP 规则

- 每个存档只有一个背包。
- 背包占用 7×9 棋盘中的一个格子，因此初始可用于食材的空间为 62 格。
- 新存档默认将背包放在右下角，但这只是初始位置，不是固定位置。
- 玩家可以把背包拖到任意空格。
- 如果目标格已有食材或其他不可覆盖物件，拖动失败，背包回到原格。
- 点击而非拖动背包时，打开背包界面。MVP 尚未实现背包内容界面时，允许先显示占位反馈。
- 背包不能合成、出售、删除或被食材生成逻辑覆盖。
- 自动整理只整理普通合成物品，不移动背包。
- 棋盘底部删除独立的“背包”按钮，只保留“食材篮”“返回食肆”和“菜单”。

## 数据设计

背包属于棋盘特殊物件，不作为普通 `ItemId` 参与合成链。游戏状态保存唯一的背包格子索引，并确保索引处不能同时保存普通食材。

加载旧存档时，如果没有背包位置，则优先使用右下角空格；右下角被占用时，从棋盘末尾向前寻找第一个空格。若棋盘没有空格，应保留存档数据并等待玩家腾出空间，不能覆盖或删除已有食材。

## 交互设计

背包格使用与普通食材明显不同的绿色功能底色和“背包”文字或图标。拖动开始后，背包跟随指针；经过可放置空格时显示落点提示；释放后提交位置变化。移动距离未超过拖动阈值时按点击处理，防止轻微手抖触发移动。

鼠标和触摸使用同一套事件路径，保证浏览器预览和微信小游戏表现一致。

## 代码边界

- 棋盘核心模块负责判断背包能否移动，并返回更新后的棋盘状态。
- 游戏状态与存档模块负责背包位置的初始化、持久化和旧存档兼容。
- `BoardSceneController` 负责绘制背包、处理点击和拖动事件，不重复实现棋盘规则。
- 普通合成逻辑继续只处理食材，不需要识别背包的合成等级。

## 异常处理

- 非法目标索引：保持原位置。
- 目标格非空：保持原位置。
- 存档中的背包索引越界或与食材冲突：按旧存档兼容规则寻找空格。
- 重复生成背包：只保留状态中记录的唯一背包，不创建第二个。

## 测试范围

- 新游戏默认生成唯一背包，并占用右下角。
- 背包可以移动到空格。
- 背包不能移动到已有食材的格子。
- 背包不能参与合成、出售或普通物品生成。
- 保存并读取后背包位置保持不变。
- 旧存档能够补充背包位置，且不会覆盖已有食材。
- 棋盘界面不再显示底部背包按钮，并能在棋盘格内渲染背包。
